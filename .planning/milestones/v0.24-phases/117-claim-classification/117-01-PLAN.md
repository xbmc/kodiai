## Objective

Create a standalone claim classification module that decomposes LLM review findings into individual claims, classifies each as `diff-grounded`, `external-knowledge`, or `inferential`, and attaches results to finding objects for downstream severity demotion (Phase 118) and output filtering (Phase 119). Uses a hybrid approach: fast heuristic pass first, LLM second-pass for ambiguous cases. Fail-open on errors.

## must_haves

- [ ] `classifyClaims()` accepts findings + diff context + PR metadata and returns findings annotated with claim classifications
- [ ] Each finding is decomposed into individual claims, each labeled `diff-grounded`, `external-knowledge`, or `inferential`
- [ ] Heuristic pass detects external-knowledge signals: version numbers, release dates, API behavior claims, behavioral assertions, CVE references, performance characteristics, compatibility claims
- [ ] LLM second-pass triggers for heuristic-ambiguous findings
- [ ] Each finding gets a summary classification: `primarily-diff-grounded`, `primarily-external`, or `mixed`
- [ ] Fail-open: classification errors/timeouts return findings unchanged (treated as diff-grounded)
- [ ] Classification results are attached to finding objects via type intersection pattern
- [ ] Integration into review pipeline after enforcement, before suppression matching

### Action

Create `src/lib/claim-classifier.ts` with:

**Types:**
```typescript
export type ClaimLabel = 'diff-grounded' | 'external-knowledge' | 'inferential';
export type SummaryLabel = 'primarily-diff-grounded' | 'primarily-external' | 'mixed';

export type ClaimClassification = {
  text: string;
  label: ClaimLabel;
  evidence?: string;
  confidence: number;
};

export type FindingClaimClassification = {
  summaryLabel: SummaryLabel;
  claims: ClaimClassification[];
};

// Input type — minimal shape the classifier needs
export type FindingForClassification = {
  commentId: number;
  filePath: string;
  title: string;
  severity: string;
  category: string;
  startLine?: number;
  endLine?: number;
};

// Output type — finding annotated with classification
export type ClaimClassifiedFinding = FindingForClassification & {
  claimClassification: FindingClaimClassification;
};

export type DiffContext = {
  rawPatch: string;  // full file diff text
  addedLines: string[];
  removedLines: string[];
  contextLines: string[];
};

export type ClassifierInput = {
  findings: FindingForClassification[];
  fileDiffs: Map<string, DiffContext>;  // filePath -> parsed diff
  prDescription: string | null;
  commitMessages: string[];
};
```

**Core functions:**

1. `extractClaims(findingTitle: string, findingBody?: string): string[]` — Split finding text into individual claim sentences. Use sentence boundary detection. Finding title is always one claim; body may contain multiple.

2. `classifyClaimHeuristic(claim: string, diffContext: DiffContext | undefined, prDescription: string | null, commitMessages: string[]): ClaimClassification` — Fast pattern-based classification:
   - **External-knowledge signals** (regex patterns):
     - Specific version numbers not in diff: `/\b\d+\.\d+\.\d+\b/` cross-referenced against diff content
     - Release date references: `/(?:introduced|released|added|available)\s+(?:in|since|from)\s+(?:v?\d|(?:January|February|March|April|May|June|July|August|September|October|November|December)\s+\d{4})/i`
     - API behavior assertions: `/(?:is known to|will always|never returns|always throws|by default .+ returns)/i`
     - Behavioral assertions about libraries: `/(?:this (?:library|package|module|framework) (?:is|has|does|will|can))/i`
     - CVE references: `/CVE-\d{4}-\d+/`
     - Performance characteristics not visible in code: `/(?:is O\([^)]+\)|has .+ complexity|scales (?:linearly|quadratically|exponentially))/i` — BUT if the claim's code pattern (e.g., nested loops) appears in the diff, classify as `diff-grounded` instead
     - Compatibility claims: `/(?:compatible with|requires .+ version|works with .+ \d)/i`
   - **Diff-grounded signals:**
     - Claim references specific line content that appears in addedLines or removedLines
     - Claim describes a change visible in the diff (e.g., "removes the null check", "adds error handling")
     - Claim quotes code that exists in the diff context
     - Claim references content from PR description or commit messages
   - **Inferential signals:**
     - Claim uses deductive language: "could cause", "may lead to", "might result in", "this means that"
     - AND the premise is visible in the diff (e.g., "removing X could cause Y" where removing X is in the diff)
   - Set confidence: 0.9+ for clear pattern matches, 0.5-0.7 for weaker signals
   - Set evidence: human-readable explanation (e.g., "version '3.2.1' not found in diff content")

3. `computeSummaryLabel(claims: ClaimClassification[]): SummaryLabel` — Aggregate logic:
   - All diff-grounded or inferential -> `primarily-diff-grounded`
   - All external-knowledge -> `primarily-external`
   - Mix -> `mixed`
   - Empty claims array -> `primarily-diff-grounded` (fail-open)

4. `classifyClaims(input: ClassifierInput): ClaimClassifiedFinding[]` — Main entry point. For each finding:
   - Extract claims from title
   - Classify each claim via heuristic
   - Compute summary label
   - Return annotated finding
   - Wrap entire function in try/catch: on error, return findings with default classification (`primarily-diff-grounded`, empty claims array)
