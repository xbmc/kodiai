Create `src/knowledge/wiki-update-generator.ts` with these exported utility functions:

**1. `matchPatchesToSection(sectionChunks: WikiPageRecord[], evidenceRows: PREvidence[]): SectionPatchMatch`**

Token-overlap matching algorithm:
- Extract tokens from section: split heading + body text on `\W+`, filter length > 3, filter out `DOMAIN_STOPWORDS` (import from wiki-staleness-detector.ts)
- For each evidence row: extract tokens from `filePath` (split on `/._-`) + `patch` content (split on `\W+`), filter same way
- Count non-stopword token overlap between section tokens and each patch's tokens
- Include patches with overlap >= 2 tokens
- Sort by heuristic_score DESC
- Cap at 5 patches
- Cap total patch content at 3000 chars (truncate last patch if needed)
- Return `SectionPatchMatch` with combined section content, matching patches, total overlap score

**2. `buildGroundedSectionPrompt(opts: { sectionHeading: string | null; sectionContent: string; patches: Array<{ prNumber: number; prTitle: string; patch: string }> }): string`**

Build the generation prompt with strict grounding rules:
```
Update this wiki section based ONLY on the code changes shown below.

SECTION: {heading or "(Lead section)"}
{sectionContent}

CODE CHANGES (from merged PRs):
### PR #{number}: {title}
```diff
{patch}
```

RULES:
1. Begin with "WHY: " followed by 1-2 sentences explaining why this section needs updating
2. Then output the COMPLETE updated section content
3. Every factual change you make MUST cite the specific PR inline, e.g., "(PR #27901)"
4. Link format: https://github.com/{owner}/{repo}/pull/{number}
5. If a change cannot be grounded in the patches above, DO NOT include it
6. If the patches show no wiki-relevant changes for this section, respond with only "NO_UPDATE"
```

The owner/repo should be parameterized (passed from options).

**3. `parseGeneratedSuggestion(text: string): { whySummary: string; suggestion: string; isNoUpdate: boolean }`**

- If text.trim() starts with "NO_UPDATE", return `{ whySummary: "", suggestion: "", isNoUpdate: true }`
- If text starts with "WHY: ", extract everything up to the first double newline as `whySummary`, rest as `suggestion`
- If text doesn't start with "WHY: ", use first sentence as whySummary, rest as suggestion (fallback)

**4. `checkGrounding(suggestionText: string, inputPrNumbers: number[]): boolean`**

- Extract all `PR #(\d+)` patterns from the suggestion text
- Check if at least one extracted PR number matches a number in `inputPrNumbers`
- Return true if at least one match, false otherwise

Create corresponding test file `src/knowledge/wiki-update-generator.test.ts` with tests for all four functions. Use the project's existing test patterns (look at wiki-staleness-detector.test.ts or wiki-voice-analyzer.test.ts for conventions).

Test data: Create realistic wiki section text about Kodi audio pipeline, realistic patch content from a PR that changes audio code, and verify the matching, prompt building, parsing, and grounding check work correctly.

<task type="auto">
  <name>Task 2: Implement createUpdateGenerator factory with full pipeline orchestration</name>
  <files>src/knowledge/wiki-update-generator.ts</files>
  <action>
Add the `createUpdateGenerator` factory function to `wiki-update-generator.ts`. This is the main orchestration function.

**`createUpdateGenerator(opts: UpdateGeneratorOptions): { run(runOpts: { topN?: number; dryRun?: boolean; pageIds?: number[] }): Promise<UpdateGeneratorResult> }`**

The `run()` method implements this pipeline:

1. **Determine pages to process:**
   - If `pageIds` provided: use those directly
   - Otherwise: Query `wiki_page_popularity` for top N pages (default 20) by composite_score DESC. Then filter to only pages that have entries in `wiki_pr_evidence` (i.e., stale pages with evidence). Use SQL JOIN:
     ```sql
     SELECT DISTINCT wpp.page_id, wpp.page_title, wpp.composite_score
     FROM wiki_page_popularity wpp
     INNER JOIN wiki_pr_evidence wpe ON wpe.matched_page_id = wpp.page_id
     ORDER BY wpp.composite_score DESC
     LIMIT ${topN}
     ```

2. **For each page (sequential with rate limiting):**
   a. Fetch all chunks via `wikiPageStore.getPageChunks(pageId)`
   b. Group chunks by `sectionHeading` into sections (Map<string | null, WikiPageRecord[]>)
   c. Fetch all PR evidence for this page:
      ```sql
      SELECT * FROM wiki_pr_evidence WHERE matched_page_id = ${pageId} ORDER BY merged_at DESC
      ```
   d. For each section: call `matchPatchesToSection(sectionChunks, evidenceRows)`
   e. Filter out sections where `matchingPatches.length === 0` (skip silently per CONTEXT.md)
   f. Build `SectionInput[]` for the voice pipeline: for each matched section, build the diff evidence string from the patches (PR number + title + patch content)

3. **Create voice-preserving pipeline** via `createVoicePreservingPipeline`:
   - `generateSectionUpdate` callback: builds grounded prompt via `buildGroundedSectionPrompt`, then calls `generateWithFallback` with `TASK_TYPES.SECTION_UPDATE`
   - Pass through `taskRouter`, `costTracker`, `logger`, `wikiPageStore`

4. **Process each page through voice pipeline** via `pipeline.processPage(pageId, sectionInputs)`

5. **Post-process each VoicePreservedUpdate:**
   - Call `parseGeneratedSuggestion` on the suggestion text
   - If `isNoUpdate`: set grounding_status = "no_update", skip storage
   - Call `checkGrounding(suggestion, inputPrNumbers)`:
     - If grounded: set grounding_status = "grounded"
     - If ungrounded: set grounding_status = "ungrounded", increment `suggestionsDropped`, skip storage (per CONTEXT.md: drop entirely)

6. **Store grounded suggestions** (unless dryRun):
   ```sql
   INSERT INTO wiki_update_suggestions (
     page_id, page_title, section_heading, original_content,
     suggestion, why_summary, grounding_status,
     citing_prs, voice_mismatch_warning, voice_scores, generated_at
   ) VALUES (...)
   ON CONFLICT (page_id, COALESCE(section_heading, '')) DO UPDATE SET
     suggestion = EXCLUDED.suggestion,
     why_summary = EXCLUDED.why_summary,
     grounding_status = EXCLUDED.grounding_status,
     citing_prs = EXCLUDED.citing_prs,
     voice_mismatch_warning = EXCLUDED.voice_mismatch_warning,
     voice_scores = EXCLUDED.voice_scores,
     generated_at = EXCLUDED.generated_at
   ```

   **Important:** Since PostgreSQL UNIQUE doesn't treat NULLs as equal, change the UNIQUE constraint approach. Instead of relying on the table UNIQUE constraint for NULL section_heading, use a functional unique index in the migration (Plan 01 creates the constraint — the executor should adjust if needed). For the ON CONFLICT, use a raw SQL approach:
   - First DELETE existing row for this (page_id, section_heading) pair
   - Then INSERT the new row
   This is simpler and avoids the NULL uniqueness issue. Wrap in a transaction.

7. **Rate limit:** `await new Promise(r => setTimeout(r, opts.rateLimitMs ?? 300))` between pages

8. **Logging:** Log at info level per page (pageTitle, sectionsMatched, suggestionsGenerated, suggestionsDropped). Log at debug level per section.

9. **Error handling:** Wrap each page in try/catch. On failure, log error at error level with page details, continue to next page (fail-open pattern per project convention).

10. **Return `UpdateGeneratorResult`** with aggregate counts and duration.

Import `generateWithFallback` from `../llm/generate.ts` and `TASK_TYPES` from `../llm/task-types.ts` for the `generateSectionUpdate` callback implementation.
