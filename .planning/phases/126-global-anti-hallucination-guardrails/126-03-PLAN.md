    - mentionAdapter.buildGroundingContext includes issue body, PR description, conversation history, retrieval results as providedContext
    - mentionAdapter.reconstructOutput removes flagged sentences and preserves markdown heading structure (orphaned headings with no content are removed)
    - mentionAdapter.minContentThreshold = 15
    - slackAdapter.extractClaims splits conversational text into sentences
    - slackAdapter.buildGroundingContext includes retrieval results and repo code context
    - slackAdapter.reconstructOutput joins kept sentences with spaces
    - slackAdapter.minContentThreshold = 5
    - troubleshootAdapter.extractClaims splits synthesized guidance into sentences
    - troubleshootAdapter.buildGroundingContext includes resolvedIssues texts + wikiMatches texts
    - troubleshootAdapter.reconstructOutput preserves bullet point structure
    - troubleshootAdapter.minContentThreshold = 20

  <action>
    All three adapters are text-based (markdown/plain text output). They share a common pattern:
    - extractClaims: reuse `extractClaims()` from `../claim-classifier.ts`
    - buildGroundingContext: collect surface-specific input texts into providedContext array
    - reconstructOutput: join kept claim sentences, handle markdown structure

    1. **mention-adapter.ts**:
       - `MentionInput`: { issueBody?: string, prDescription?: string, conversationHistory?: string[], retrievalResults?: string[], diffPatches?: string[] }
       - `MentionOutput`: string (markdown response body)
       - `extractClaims`: Use `extractClaims()` on the response text
       - `buildGroundingContext`: Flatten all input fields into providedContext. If diffPatches provided, parse first one as diffContext.
       - `reconstructOutput`: Join kept sentences. After joining, scan for markdown headings (##, ###) that have no content between them or before end -- remove orphaned headings. Preserve code blocks (``` fenced blocks) as-is (never filter inside code blocks).
       - `surface`: "mention", `minContentThreshold`: 15

    2. **slack-adapter.ts**:
       - `SlackInput`: { retrievalResults?: string[], repoContext?: string[], userMessage: string }
       - `SlackOutput`: string (answer text)
       - `extractClaims`: Use `extractClaims()` on answer text
       - `buildGroundingContext`: retrievalResults + repoContext + userMessage as providedContext
       - `reconstructOutput`: Join kept sentences with spaces, trim
       - `surface`: "slack", `minContentThreshold`: 5

    3. **troubleshoot-adapter.ts**:
       - `TroubleshootInput`: { resolvedIssues: Array<{title: string, body: string, tailComments: string[], semanticComments: string[]}>, wikiResults: Array<{pageTitle: string, rawText: string}>, issueTitle: string, issueBody: string | null }
       - `TroubleshootOutput`: string (synthesized guidance)
       - `extractClaims`: Use `extractClaims()` on guidance text
       - `buildGroundingContext`: Flatten resolved issue bodies, comments, wiki texts, issue title/body into providedContext
       - `reconstructOutput`: Join kept sentences, preserve bullet points (lines starting with - or *)
       - `surface`: "troubleshoot", `minContentThreshold`: 20
  </action>

  <behavior>

    - triageAdapter.buildGroundingContext includes issue title, body, and label descriptions
    - triageAdapter.minContentThreshold = 10
    - wikiAdapter.extractClaims extracts sentences from wiki update suggestion text
    - wikiAdapter.buildGroundingContext uses PR evidence patches (diff content) and wiki page content as providedContext
    - wikiAdapter.reconstructOutput preserves MediaWiki template markers ({{...}}) and heading structure
    - wikiAdapter.minContentThreshold = 10
    - wikiAdapter replaces the role of checkGrounding() in wiki-update-generator.ts

  <action>
    1. **triage-adapter.ts**:
       - `TriageInput`: { issueTitle: string, issueBody: string | null, labelDescriptions?: string[] }
       - `TriageOutput`: string (triage comment body)
       - `extractClaims`: Use `extractClaims()` on the prose sections. Skip lines that are pure template (matching table rows `| ... |`, or HTML tags like `<details>`). Only classify free-text prose.
       - `buildGroundingContext`: issueTitle + issueBody + labelDescriptions as providedContext
       - `reconstructOutput`: Join kept sentences, preserve table/HTML structure unchanged
       - `surface`: "triage", `minContentThreshold`: 10

    2. **wiki-adapter.ts**:
       - `WikiInput`: { patchDiffs: Array<{prNumber: number, prTitle: string, patch: string}>, wikiPageContent: string, wikiPageTitle: string }
       - `WikiOutput`: string (suggestion text with PR citations)
       - `extractClaims`: Use `extractClaims()` on suggestion text
       - `buildGroundingContext`: Parse patchDiffs into providedContext strings (include PR numbers so citation references can be grounded). Include wikiPageContent as context. Set contextSources to patch PR numbers.
       - `reconstructOutput`: Join kept sentences. Preserve {{template}} markers -- if a sentence contains a {{...}} template, always keep it regardless of classification. Preserve heading structure.
       - `surface`: "wiki", `minContentThreshold`: 10

       The wiki adapter effectively replaces `checkGrounding()` from wiki-update-generator.ts. Instead of a binary "has PR citation" check, the unified pipeline does sentence-level classification and surgical removal. The grounding is richer: PR patches become providedContext, so claims about code changes can be verified against actual diffs.
