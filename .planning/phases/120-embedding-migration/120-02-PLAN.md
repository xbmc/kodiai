<task type="auto">
  <name>Task 1: Create wiki embedding backfill script for voyage-context-3 migration</name>
  <files>scripts/wiki-embedding-backfill.ts</files>
  <action>
Create `scripts/wiki-embedding-backfill.ts` -- a standalone CLI script that re-embeds all wiki page chunks with voyage-context-3 using `contextualizedEmbed()`.

**CLI interface** (follow scripts/backfill-wiki.ts conventions):
```
Usage: bun scripts/wiki-embedding-backfill.ts [options]

Options:
  --model <name>    Target embedding model (default: voyage-context-3)
  --delay <ms>      Delay between pages in ms (default: 500)
  --dry-run         Calculate totals without writing
  --help            Show help
```

**Environment:** `DATABASE_URL` (required), `VOYAGE_API_KEY` (required)

**Algorithm:**

1. **Pre-flight:** Query `SELECT COUNT(*), SUM(token_count) FROM wiki_pages WHERE deleted = false` and log total pages/tokens/estimated cost ($0.18/1M tokens, first 200M free).

2. **Group chunks by page:** Query `SELECT DISTINCT page_id FROM wiki_pages WHERE deleted = false ORDER BY page_id`. For each page_id, load all chunks for that page ordered by chunk_index.

3. **Embed per page:** For each page, collect all chunk texts into an array and call `contextualizedEmbedChunks()` from `src/knowledge/embeddings.ts`. This sends all chunks as one document: `inputs: [chunkTexts]` so chunks share context awareness from the full page.

4. **Token limit handling:** If a page's chunks exceed the 120K token limit or 32K context limit, the SDK will return an error. Catch this and fall back to embedding chunks individually using the single-text `createContextualizedEmbeddingProvider().generate()` method. Log a warning.

5. **Update in place:** For each chunk where embedding was generated, run:
   ```sql
   UPDATE wiki_pages
   SET embedding = $vector::vector,
       embedding_model = $model,
       stale = false
   WHERE id = $chunkId
   ```
   Per user decision: overwrite in place, no shadow column, no backup.

6. **Progress tracking:** Log progress every 10 pages. Log total pages, chunks embedded, chunks failed, elapsed time.

7. **Post-backfill verification:** Query `SELECT COUNT(*) FROM wiki_pages WHERE embedding_model != $model AND deleted = false AND embedding IS NOT NULL`. If > 0, log a WARNING with the count. Success criterion: zero rows with old model.

8. **HNSW index rebuild reminder:** At completion, log: "Post-migration: consider running REINDEX INDEX idx_wiki_pages_embedding_hnsw for optimal search quality."

9. **Sequential processing:** Process one page at a time with configurable delay between pages. No parallelism -- simple and rate-limit-safe.

**Imports:**
- `VoyageAIClient` and `contextualizedEmbedChunks` from `src/knowledge/embeddings.ts`
- `createDbClient` from `src/db/client.ts`
- `runMigrations` from `src/db/migrate.ts`
- `parseArgs` from `node:util`
- `pino` for logger

**Do NOT** use the existing `backfillWikiPages()` function from wiki-backfill.ts -- that function fetches from MediaWiki API. This script only re-embeds existing stored chunks.

<task type="auto">
  <name>Task 2: Create embedding comparison benchmark script</name>
  <files>scripts/embedding-comparison.ts</files>
  <action>
Create `scripts/embedding-comparison.ts` -- a reusable benchmark that compares retrieval results between two embedding models.

**CLI interface:**
```
Usage: bun scripts/embedding-comparison.ts [options]

Options:
  --old-model <name>   Old embedding model (default: voyage-code-3)
  --new-model <name>   New embedding model (default: voyage-context-3)
  --top-k <n>          Results per query (default: 5)
  --output <path>      JSON output path (default: data/embedding-comparison.json)
  --help               Show help
```

**Environment:** `DATABASE_URL` (required), `VOYAGE_API_KEY` (required)

**Algorithm:**

1. **Query set:** Define a hardcoded set of 10-15 eval queries representative of real wiki search use cases. Examples:
   - "How to install Kodi on Windows"
   - "PVR setup with TVHeadend"
   - "Audio passthrough configuration"
   - "Kodi Python addon development"
   - "Skin development XML structure"
   - "MySQL shared database setup"
   - "HDR video playback settings"
   - "Kodi remote control setup"
   - "Log file location and debugging"
   - "Add-on repository configuration"
   (Claude's discretion per user decision -- these are reasonable kodi.wiki queries)

2. **For each query:**
   - Generate embedding with old model using `createEmbeddingProvider({ model: oldModel })`
   - Generate embedding with new model using `createContextualizedEmbeddingProvider({ model: newModel })`
   - Run `store.searchByEmbedding()` with each embedding, topK results
   - Record: query, old results (title, distance, section), new results (title, distance, section)

3. **Console output:** Print a formatted table for each query:
   ```
   Query: "How to install Kodi on Windows"

   Old (voyage-code-3)              | New (voyage-context-3)
   ─────────────────────────────────┼────────────────────────────────
   1. Installing Kodi (0.32)        | 1. Installing Kodi (0.28)
   2. Windows FAQ (0.45)            | 2. Windows FAQ (0.31)
   ...
   ```

4. **Summary stats:** After all queries, print:
   - Average distance improvement (new avg vs old avg)
   - Number of queries where top-1 result changed
   - Number of queries where new distances are lower (better)

5. **JSON output:** Write detailed results to the output path for analysis:
   ```json
   {
     "timestamp": "...",
     "oldModel": "voyage-code-3",
     "newModel": "voyage-context-3",
     "queries": [
       {
         "query": "...",
         "oldResults": [{ "pageTitle": "...", "distance": 0.32, "section": "..." }],
         "newResults": [{ "pageTitle": "...", "distance": 0.28, "section": "..." }]
       }
     ],
     "summary": {
       "avgDistanceOld": 0.42,
       "avgDistanceNew": 0.35,
       "improvement": "16.7%",
       "top1Changed": 3,
       "queriesImproved": 8
     }
   }
   ```

6. **Reusability:** Per user decision, this script is kept for future model evaluations (voyage-context-4, etc.). The --old-model and --new-model flags make it model-agnostic. The query set is a const array that can be extended.

**Important:** This script should work BEFORE the backfill (comparing current embeddings stored in DB against fresh embeddings from the new model). It generates both old and new embeddings on the fly for the query side, then searches the existing DB embeddings. This means:
- Before backfill: DB has voyage-code-3 embeddings, so "old" search works natively. "New" search will show what results WOULD look like if DB were re-embedded (since query embedding is new model but DB vectors are old model, the new results will be degraded -- note this in output).
- After backfill: Both work correctly since DB now has voyage-context-3 embeddings.

- `runMigrations` from `src/db/migrate.ts`
- `parseArgs` from `node:util`, `pino`, `fs/promises`
