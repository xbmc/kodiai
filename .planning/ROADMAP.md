## Phase Details

### Phase 120: Embedding Migration
**Goal**: Wiki corpus uses voyage-context-3 embeddings and retrieval pipeline routes queries to the correct model per corpus
**Depends on**: Nothing (first phase, independent of other v0.25 work)
**Requirements**: EMBED-01, EMBED-02, EMBED-03, EMBED-04
**Success Criteria** (what must be TRUE):
  1. All wiki page embeddings in PostgreSQL are generated by voyage-context-3 (zero rows with voyage-code-3 embeddings remain in wiki_pages)
  2. Wiki retrieval queries use voyage-context-3 for query embedding while all other corpora (code, reviews, issues, snippets) continue using voyage-code-3
  3. wiki-store.ts accepts embedding model as a parameter instead of hardcoding "voyage-code-3"
  4. Future wiki page ingestion (sync) automatically uses voyage-context-3 via the contextualized embedding API
**Plans**: 2 plans

Plans:
- [ ] 120-01-PLAN.md — Contextualized embedding provider, wiki-store parameterization, and per-corpus routing
- [ ] 120-02-PLAN.md — Wiki embedding backfill script and comparison benchmark

### Phase 121: Page Popularity
**Goal**: Every wiki page has a composite popularity score combining inbound links, citation frequency, and edit recency, enabling principled top-N page selection
**Depends on**: Phase 120 (retrieval must use correct embeddings before citation tracking is meaningful)
**Requirements**: POP-01, POP-02, POP-03, POP-04
**Success Criteria** (what must be TRUE):
  1. Inbound link counts from MediaWiki linkshere API are stored for all wiki pages
  2. Retrieval citation frequency is tracked -- how often each wiki page appears in retrieval results is counted and persisted
  3. A composite popularity score exists per page combining inbound links, citation frequency, and edit recency into a single ranked list
  4. The pipeline can request the top-N pages by popularity score and receive a deterministic ordered result
**Plans**: 2 plans

Plans:
- [ ] 121-01-PLAN.md — Schema, store, config, and retrieval pipeline citation instrumentation
- [ ] 121-02-PLAN.md — Linkshere fetcher, composite scorer with scheduler, and backfill script

### Phase 122: Enhanced Staleness
**Goal**: Staleness detection is grounded in actual code changes from recent merged PRs, with diff content preserved as evidence for the update generator
**Depends on**: Phase 121 (popularity scores drive which pages to evaluate for staleness)
**Requirements**: STALE-01, STALE-02, STALE-03, STALE-04
**Success Criteria** (what must be TRUE):
  1. Merged PRs from the last 90 days are scanned and their changed code areas identified
  2. Changed code areas are matched to related wiki pages via the retrieval pipeline
  3. Diff content (patches) from PRs/commits is preserved and available to downstream consumers (not discarded as in the current staleness detector)
  4. False positive rate is reduced via domain stopwords and section-heading weighting (common tokens like "player" and "video" no longer match every wiki page)

### Phase 123: Update Generation
**Goal**: LLM generates section-level rewrite suggestions for stale wiki pages, grounded in actual diff content with commit/PR citations
**Depends on**: Phase 122 (requires diff-grounded staleness evidence as input)
**Requirements**: UPDATE-01, UPDATE-02, UPDATE-03, UPDATE-04
**Success Criteria** (what must be TRUE):
  1. For each stale wiki page, the LLM produces section-level rewrite suggestions (not full-page rewrites)
  2. Every suggestion is grounded in actual code diff content and cites the specific PR(s) or commit(s) that motivated the change
  3. Suggestions that cannot be grounded in verified diff content are classified as "inferred" and excluded from publishing
  4. The top 20 pages by composite popularity score are processed per pipeline run

### Phase 124: Publishing
**Goal**: Update suggestions are posted as structured comments on a tracking issue in xbmc/wiki with rate-limit safety
**Depends on**: Phase 123 (requires generated suggestions to publish)
**Requirements**: PUB-01, PUB-02, PUB-03, PUB-04
**Success Criteria** (what must be TRUE):
  1. A tracking issue is created in xbmc/wiki with a batch summary of all pages evaluated
  2. Each page's update suggestions are posted as an individual comment on the tracking issue
  3. Comment posting respects GitHub secondary rate limits with minimum delays and backoff on 403 responses
  4. GitHub App installation on xbmc/wiki is verified before any publishing attempt (pre-flight check fails gracefully with actionable error)
